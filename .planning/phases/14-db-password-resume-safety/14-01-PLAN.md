<objective>
Prevent DB_PASSWORD mismatch on resume installation when PostgreSQL data already exists.

Purpose: When `install.sh` resumes from a checkpoint (e.g., phase 5+), phase 4 (Configuration) re-runs `generate_config()` which regenerates ALL secrets. If PG already has data with the old password, the new password causes `FATAL: password authentication failed` and the stack cannot start. The fix preserves secrets from the `.env.backup.*` file when PG data is detected.

Output: Modified `lib/config.sh` with secret preservation logic, hardened `lib/compose.sh:sync_db_password()` with longer timeout and better error messages.
</objective>

1. Checks if PG data directory exists: `"${INSTALL_DIR}/docker/volumes/db/data/PG_VERSION"` (PG_VERSION file is the definitive indicator of initialized PostgreSQL data — more reliable than checking if the directory simply exists, because `_create_directory_structure()` creates `volumes/db/data` unconditionally at line 71).

2. If PG data exists, finds the most recent `.env.backup.*` file:
   ```bash
   local latest_backup
   latest_backup="$(ls -t "${INSTALL_DIR}/docker/.env.backup."* 2>/dev/null | head -1)"
   ```

3. If backup found, extracts and restores these three critical secrets:
   - `DB_PASSWORD` — PostgreSQL authentication
   - `REDIS_PASSWORD` — Redis ACL authentication
   - `SECRET_KEY` — Dify encryption key for stored data

   Extraction pattern for each:
   ```bash
   local saved_val
   saved_val="$(grep "^VAR_NAME=" "$latest_backup" | cut -d'=' -f2-)"
   if [[ -n "$saved_val" ]]; then
       _VAR_NAME="$saved_val"
       restored=true
   fi
   ```

4. Returns 0 if any secrets were restored (caller should log), 1 if nothing restored.

5. If PG data exists but NO backup found: log `log_warn "PG data exists but no .env backup found — generating new password (sync_db_password will fix)"` and return 1.

Then modify `_generate_secrets()` to call `_restore_secrets_from_backup` AFTER generating all fresh secrets. The flow becomes:
```bash
_generate_secrets() {
    # Generate fresh secrets (existing code, unchanged)
    _SECRET_KEY="$(generate_random 64)"
    _DB_PASSWORD="$(generate_random 32)"
    _REDIS_PASSWORD="$(generate_random 32)"
    # ... all other secrets unchanged ...

    # Override volume-bound secrets from backup if PG data exists (IREL-03)
    if _restore_secrets_from_backup; then
        log_info "Restored DB_PASSWORD, REDIS_PASSWORD, SECRET_KEY from previous installation"
    fi

    # Fatal check (existing code, unchanged)
    if [[ -z "$_SECRET_KEY" || -z "$_DB_PASSWORD" || ... ]]; then
        ...
    fi

    # Export (existing code, unchanged)
    ...
}
```

This approach is safe because:
- Fresh secrets are ALWAYS generated first (fallback if restore fails)
- `_restore_secrets_from_backup` only OVERWRITES the three volume-bound variables
- Other secrets (SANDBOX_API_KEY, PLUGIN_DAEMON_KEY, etc.) are always fresh — they are not persisted in volumes
- The `set -euo pipefail` is respected: `_restore_secrets_from_backup` returns cleanly on all paths
- On fresh install: PG_VERSION does not exist, function returns 1 immediately, no change in behavior
