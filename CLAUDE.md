## 1. Общие правила

- Всегда отвечай **на русском языке**.
- Работай в тесной связке с **GSD** как основным workflow:
  `discuss → plan → execute → verify`.
- Уважай существующие артефакты планирования:
  `.planning/PROJECT.md`, `.planning/REQUIREMENTS.md`,
  `.planning/ROADMAP.md`, `.planning/STATE.md`, `TASKS.md`,
  `CLAUDE_CODE_DRIVER.md`. Не затирай и не пересоздавай их
  без явной причины и моего подтверждения.
- Если видишь расхождение между фактическим кодом и планом в
  `.planning/*` — сначала предложи, как синхронизировать
  (обновить план или скорректировать код).

## 2. Git и деплой

- Никогда **не выполняй** `git add / commit / push`, `rebase`,
  `reset`, `push --force` без моего прямого текстового запроса.
- Считай, что CI/деплоер уже подключён к GitHub. Твоя задача:
  после каждой фазы проект должен:
  - собираться без ошибок;
  - подниматься через `install.sh` / соответствующие скрипты;
  - не ломать существующие профили деплоя.
- При изменениях, влияющих на деплой (Dockerfile, docker-compose,
  install.sh, scripts/*), всегда:
  1) опиши план изменений;
  2) дождись подтверждения;
  3) в конце предложи перечень команд для локальной проверки.

## 3. Стиль работы с кодом

- Отдавай предпочтение **простым и очевидным решениям**.
- Поддерживай читаемость:
  - функции и скрипты короткие, делают одну вещь;
  - явные названия переменных;
  - минимум «магических» констант, выноси в конфиги.
- При изменении **критичного кода** (деплой, security, credentials,
  сетевые настройки) всегда:
  - обновляй или добавляй тесты / проверки;
  - описывай, как воспроизвести и проверить поведение.
- При крупных изменениях:
  - сначала предложи high-level план;
  - дождись моего подтверждения;
  - только потом переходи к реализации.

## 4. Взаимодействие с GSD

- Перед любой gsd-командой используй `/clear` для свежего контекста.
- Для каждой фазы N придерживайся цикла:
  1) `/gsd:discuss-phase N` — если нужны дизайн-решения;
  2) `/gsd:plan-phase N` — формируем детальный план;
  3) `/gsd:execute-phase N` — выполняем план;
  4) `/gsd:verify-work N` — проверяем достижение целей.
- Если возникают **API Error 500** или проблемы с субагентами:
  - отключай сабагентов и /loop;
  - планируй и выполняй фазу в одном контексте,
    опираясь на `.planning/PLAN*.md` и `STATE.md`.
- Всегда синхронизируй результат работы с
  `.planning/STATE.md` и соответствующими SUMMARY/PLAN
  файлами, чтобы следующий запуск GSD мог продолжить
  с актуального состояния.

## 5. Безопасность

- Никогда не хардкодь credentials (IP, пароли, токены, ключи)
  в код, конфиги по умолчанию, README или другие публичные файлы.
- Credentials должны храниться только в безопасных местах
  (например, `credentials.txt` с `chmod 600`) и использоваться
  через переменные окружения или секреты.
- Не выводи чувствительные данные в stdout, логи, UI-подсказки.
- При изменениях security-частей (firewall, rate limiting,
  auth, TLS, SSH, VPN):
  - явно описывай угрозы, которые мы закрываем;
  - не ослабляй безопасность ради удобства без моего согласия.

## 6. Проект AgMind (контекст)

- Это установщик корпоративной RAG-платформы:
  **Dify + Open WebUI + Ollama/vLLM + Weaviate/Qdrant + мониторинг**,
  всё в Docker Compose.
- Одна команда `sudo bash install.sh` поднимает полный стек
  (~23-34 контейнера) с разными профилями:
  **LAN / VPN / VPS / Offline**.
- Основной shell — **Bash 5+** c строгими настройками:
  `set -euo pipefail`, скрипты должны проходить `shellcheck`.
- Все Docker-образы привязаны к конкретным версиям через
  `versions.env`. Запрещено использовать `:latest` или
  нестабильные тэги.
- При изменениях, влияющих на профили деплоя (LAN/VPN/VPS/Offline),
  обязательно указывай, какие профили затронуты и как их тестировать.

## 7. Локальные правила для будущих улучшений

- Если видишь архитектурный долг (повтор кода, хрупкие места),
  сначала зафиксируй его в `.planning/REQUIREMENTS.md` или
  `TASKS.md` как отдельную задачу, а не чини «по пути».
- Предпочитай **миграции без даунтайма** там, где это возможно:
  новые версии сервисов, изменения схемы данных, перестройка пайплайнов.
